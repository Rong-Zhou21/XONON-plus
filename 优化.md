● Memento 的核心创新点详解                                                         
                                                                              
  下面把 Memento 的方法学拆成五条主线，每一条都给出"它解决了什么问题、它的设计是什 
  么、设计的优劣权衡"，便于你独立判断哪些值得继承、哪些要重构。
                                                                                   
  ---                                                             
  一、问题重构：把持续学习从「梯度更新」重新表述为「记忆增广的 MDP」               
                                                                                   
  它解决的问题                                                                     
                                                                                   
  传统 LLM agent 的持续学习路线是 RLHF / SFT / DPO 这类——靠梯度更新让模型本身记住经
  验。这条路有三个硬伤：成本极高、知识不可解释、容易遗忘旧能力。                   
                                                                                   
  Memento 的设计                                                  

  把"持续学习"重新形式化为一个 memory-augmented MDP：                              
   
  - 状态 s 不再只是当下任务上下文，而是 (当前任务上下文, 当前案例库 M)             
  - 动作 a 包含两部分：对环境的动作 + 对案例库 M 的读写动作       
  - 智能体的策略不再是"如何回答问题"，而是"如何在案例库中读、如何在案例库中写"     
                                                                                   
  也就是说，把"learn"从模型权重里搬到外部存储里，learn                             
  这个动词的对象从"参数"变成了"案例"。                                             
                                                                                   
  设计的优劣                                                                       
   
  - 优：彻底绕开梯度更新，改一次案例库立刻见效；知识可解释（你能直接读出案例库里存 
  了什么）；不存在灾难性遗忘                                      
  - 局限：上限被 LLM 的 in-context 能力卡死——案例库再丰富，LLM                     
  不会"用"也是白搭；案例库的容量管理本身又成了一个新问题                           
   
  可以独立思考的方向                                                               
                                                                  
  这个理论框架最有价值的是它把"学习"剥离成一个可独立设计的子系统。当你做调度层 CBD 
  时，问题应当类似地重新形式化为：调度器的状态空间、动作空间、案例的读写动作分别是
  什么？                                                                           
                                                                  
  ---
  二、案例的「终态压缩」：只存 (s_T, a_T, r_T)，不存完整轨迹
                                                                                   
  它解决的问题
                                                                                   
  经典 case-based reasoning 文献里，案例往往是完整的问题求解过程。但 LLM agent     
  的轨迹可以非常长（GAIA 里一次 trial
  可能有几十次工具调用），完整存下来会让案例库爆炸、检索成本指数级上升。           
                                                                  
  Memento 的设计                                                                   
   
  只保留三件事：终态 s_T（任务结束时的最终上下文）、动作                           
  a_T（最终给出的答案/计划）、奖励 r_T（成功/失败标签）。中间过程统统丢掉。
                                                                                   
  具体落到 memory.jsonl 上就是每行一个 (query, case, plan, label)。case 是对 s_T   
  的自然语言摘要，plan 是 a_T，label 是 r_T。
                                                                                   
  设计的优劣                                                                       
   
  - 优：案例库极其紧凑，万级条目无压力；检索可以用句子模型秒级完成；写入轻量       
  - 代价：丢失了"过程信息"——同样是成功，怎么成功的（中间走了哪些弯路、试了哪些工具
  ）信息全没了。这对 QA                                                            
  任务影响不大（只关心最终答案），但对控制类任务影响很大（同样拿到
  logs，砍法不同复用价值就不同）                                                   
                                                                  
  可以独立思考的方向                                                               
   
  这是 Memento 最值得你警惕的设计假设。它默认"过程不重要，终态足够"——这个假设在 QA 
  上成立，但在你要做的"调度层                                     
  CBD"上很可能不成立。如果你要继承这个压缩思想，就必须重新定义"什么是终态"——也许   
  CBD 里的"终态"是一段调度行为序列，而不是单个动作。              

  ---
  三、Planner–Executor 双层架构 + CBR 严格嵌在 Planner 端
                                                                                   
  它解决的问题
                                                                                   
  LLM agent 同时承担"想清楚做什么"和"具体怎么做"两件事时，决策质量会被工具调用细节 
  拖累。把这两件事拆开是 ReAct
  之后的常见做法，但拆开之后"经验"该归谁、给谁用，没有定论。                       
                                                                  
  Memento 的设计                                                                   
   
  - Meta-Planner（GPT-4.1）：负责把高层 query 拆成可执行子任务序列                 
  - Executor（o3 或其他）：负责把每个子任务通过 MCP 工具实现      
  - CBR 只接在 Planner 上：检索到的 case 喂给 Planner 改进它的拆分；Executor       
  不直接消费 case                                                                  
                                                                                   
  设计的优劣                                                                       
                                                                  
  - 优：架构干净，CBR 只动一处；Planner                                            
  的产出空间（任务拆分）天然适合"借鉴过去类似任务怎么拆"          
  - 隐含假设：Executor 是"无记忆的、对工具熟练的、当前可靠"的——这个假设在 GAIA     
  里大致成立（工具调用是确定性的），但在你 XENON 里完全不成立（STEVE-1             
  在不同环境下表现差异巨大）
                                                                                   
  可以独立思考的方向                                                               
   
  这是 Memento 论文跟你的诉求最不对齐的一点。Memento 默认问题在 Planner            
  端、Executor 端没事；你的判断是 XENON 的问题恰恰在 Executor 端（更准确说在"调度
  Executor"的那一层）。如果你接受这个判断，那么 Memento 在"CBR                     
  该接在哪一层"这件事上的答案对你没有借鉴价值——你需要重新设计"CBD 
  接在哪一层"，而且必须做出与 Memento 不同的回答。

  ---
  四、双阶段学习方法学：非参数 CBR → 参数化 CBR
                                                                                   
  这是 Memento 在方法学上最完整的贡献，也是最有教学价值的一条。
                                                                                   
  阶段一：非参数化 CBR（np_memory.py）                            
                                                                                   
  - 检索方式：把 case 字段过一个预训练句向量模型（默认用 SimCSE-RoBERTa 的 [CLS] 或
   pooler_output），L2 归一化后做 cosine 相似度，TopK
  - 训练成本：零。完全是无监督的相似度检索                                         
  - 作用：冷启动期的兜底方案。案例库刚建立、样本不够时，用这个先把系统跑起来，开始
  攒数据                                                                           
  - 本质：跟传统的 RAG 检索没有区别，价值不在算法本身，而在它作为参数化版本的 
  fallback 和数据收集器存在                                                        
                                                                  
  阶段二：参数化 CBR（train_memory_retriever.py + parametric_memory.py）           
                                                                  
  - 核心模型：MemoryRetrieverClassifier —— SimCSE backbone + 一个二分类头 Linear(2H
   → 512 → ReLU → Dropout → Linear → 2)                           
  - 输入形态：双塔结构。query（自然语言）和 case+plan（拼成 ICL 文本，用 [CASE] 和 
  [PLAN] 作为 section token 分隔）各过一次 backbone，分别取 [CLS] hidden，把两个 H 
  维向量 concat 成 2H，再过分类头
  - 输出：二分类 logits，softmax 后的"正类概率"就是这条 case 对当前 query          
  的"有用度分数"                                                                   
  - 训练目标：把 (query, case) 配对当二分类样本，truth_label=1 是"这条 case 帮
  planner 给出了对的计划"，truth_label=0 是"这条 case 没帮上"                      
  - 损失函数：带类别权重的 CrossEntropy。当正负样本不平衡时，自动 w_pos = 
  neg/pos，把少数类拉回来                                                          
  - 训练细节：AdamW + warmup 线性衰减 + 混合精度 + 梯度裁剪，标准的 BERT-finetune
  套路；分层切分保证 train/val 中正负比一致；评估指标用 AUC > F1 > acc 的优先级    
  - 检索时的用法：候选 case 池过一遍模型，对每条算出"有用度概率"，按概率排序取
  TopK——这是用一个学到的策略替代 cosine                                            
                                                                  
  这套学习方法学的妙处                                                             
                                                                  
  它打通了一个完整的 continual learning 闭环：                                     
   
  跑任务 → 收集 (query, case, plan, outcome) →                                     
  truth_label 由环境/评估器自动给 →                               
  攒到一定量就训一版检索器 →                                                       
  新检索器替换旧的 →                                                               
  跑下一批任务（用新检索器） → 又收集新数据 → ...                                  
                                                                                   
  关键设计：训练数据 (training_data.jsonl) 和检索池 (memory.jsonl)                 
  分离。训练数据是带 label 的样本对（包含正负），检索池是无 label                  
  的候选案例。这两份文件可以独立维护、独立扩展。                                   
                                                                  
  设计的优劣

  - 优：阶段化推进，第一天就能上线；监督信号自动产生不需要人工标注；模型轻量（SimCS
  E-base 只有 110M），CPU 都能推理；训练快（几千条数据十几分钟搞定）
  - 局限：                                                                         
    - 双塔分别 encode 后 concat 的设计没有显式建模 query-case                      
  的交互。更先进的做法是 cross-encoder（query 和 case 拼起来一起过                 
  backbone），但成本翻倍。Memento 选了双塔是为了批量推理时只需要 encode query      
  一次，case 池的 embedding 可以预计算和缓存                                       
    - 二分类目标是粗的——"有用 /                                   
  无用"，但实际上"有多有用"是连续值。可以做回归或排序学习                          
    - 训练数据的标注完全依赖"这条 case 是否被用过且效果好"这种 indirect
  信号，因果链很长——一条 case 没用，可能是因为 case 本身不好，也可能是因为 LLM     
  没看懂、也可能是因为环境太难。Memento 没有解决这种 credit assignment 问题
                                                                                   
  可以独立思考的方向                                              

  这套两阶段方法学是 Memento 最值得整体继承的部分。但你要做的 CBD                  
  在"参数化什么"这件事上有重新选择的空间：
                                                                                   
  - Memento 参数化的是"case 选择策略"——给定 query 和候选 case，预测"这条 case      
  有没有用"
  - 你做调度层 CBD                                                                 
  时，可以参数化的不只是"案例选择"，还可以是"调度行为生成"——给定情境和检索到的     
  cases，直接预测"该选哪种调度模式"。这就是你前面认可的 C 路线
                                                                                   
  这两种参数化的对象是不一样的——选择 vs 生成，前者输出 K 维                        
  softmax，后者输出动作分布。你需要决定本次 milestone 走哪条。
                                                                                   
  ---                                                             
  五、监督信号的自动闭环：不需要人工标注
                                                                                   
  它解决的问题
                                                                                   
  所有"持续学习"系统的死穴是数据从哪儿来。如果每条 case 都要人标                   
  label，系统就跑不久。                                                            
                                                                                   
  Memento 的设计                                                  

  利用任务本身有 ground-truth answer 这件事——GAIA / SimpleQA / HLE 都是有标准答案的
   benchmark，每次 trial 完了直接对答案，对了 truth_label=1、错了
  truth_label=0。整个 loop 不需要任何人工介入。                                    
                                                                  
  设计的优劣                                                                       
   
  - 优：闭环成立，越跑越多数据，越多数据训出越好的检索器，越好的检索器又跑出更多正 
  样本——是个正反馈                                                
  - 隐含假设：任务必须有客观可判定的成败信号。这在 QA、有 oracle 的 RL             
  环境里成立，在开放域对话、创意写作里就不成立                                     
   
  在你 XENON 上的对应                                                              
                                                                  
  这一点对你完全成立而且更强——Minecraft 环境给的成败信号比 QA 还硬：waypoint       
  物品有没有出现在库存里、整个 task 配方是否完成，都是程序判定的。你甚至比 Memento
  多了：                                                                           
                                                                  
  - 步数（量化"多有效"）
  - 中间状态（每一步的库存变化、视觉变化）
  - 干预次数（量化"调度器多笨"）                                                   
                                                                                   
  这些都是 Memento 用不了的、你能用的额外监督信号。这是你 CBD 系统设计上比 Memento 
  更宽的"信号带宽"。                                                               
                                                                                   
  ---                                                             
  六、Memento 的方法学边界：它没解决的问题
                                                                                   
  把 Memento 的创新点列完，反过来也要清楚它的边界。这些是它没碰过、你做 CBD
  要重新回答的问题：                                                               
                                                                  
  1. 过程信息的价值——它默认终态足够，没考虑"怎么走到终态的"是否有独立价值          
  2. 动态环境下的案例衰减——它默认 case 一旦正确就永远正确，没有"知识漂移"机制
  3. 多模态案例——它的 case 是纯文本，README 里把多模态当 TODO，没动手做            
  4. 细粒度 credit assignment——case 没被用对，到底是 case 烂还是 LLM 烂，它分不清  
  5. 生成式 case 利用——它只做"选 case"，没做"基于 case 生成新方案"                 
  6. 执行器侧的经验利用——它默认 Executor 不需要经验，只 Planner 需要               
                                                                                   
  这六个点都是潜在的论文增量空间。你做的"调度层 CBD"如果想立得住，至少要在         
  1、2、3、6 这四个点上给出和 Memento 不同的答案——而这四个点恰好对应你 XENON       
  任务的天然属性（连续控制、配方漂移、视觉观察、执行器是真正瓶颈）。

  案例库与决策器联合设计：四个候选方向                                             
                                               
  你点出的难题很关键——案例库的设计目标和决策器的训练目标必须从一开始就联动，否则就 
  会出现"案例库存了一堆漂亮的东西但决策器学不到"或"决策器要的信号案例库提供不了"的
  悲剧。下面给你四个范式层面的候选方向，每个方向把"案例怎么存"和"决策器怎么训"绑在 
  一起讲，你看哪个范式跟你的研究目标最对齐，我们再深入。                           

  四个方向的核心区别在于：案例库和决策器之间是什么样的耦合关系。                   
                                                                                   
  ---                                                                              
  方向 1：Retrieval-Decision（检索式决策）                        
                                          
  案例库的形态
                                                                                   
  每条 case 是一个三元组：(情境特征, 调度模式标签,                                 
  结果)。关键约束：调度模式必须能被离散归类（比如把所有 STEVE-1 使唤方式抽象成 N   
  个有限的"模式"，例如"单 prompt 长跑"、"两阶段切换"、"探索→采集双段式"等）。      
                                                                  
  决策器是什么

  case 选择器——给定当前情境，从案例库里检索 TopK 案例，直接套用得分最高那条 case   
  的"调度模式标签"。决策器的输出是 case 的索引（或案例 id），不是直接的调度行为。
                                                                                   
  训练目标                                                        

  跟 Memento 的二分类思路一致：(情境, case) 配对，标签是"用这条 case               
  的调度模式能否成功"。
                                                                                   
  案例库与决策器的耦合                                                             
   
  强耦合 + 弱抽象。案例库直接是决策器的"检索字典"，决策器学的是"哪条 case          
  适合当前情境"。运行时一定要查案例库。                           
                                                                                   
  关键挑战                                                                         
   
  - 调度模式必须能枚举出来（你得先做一次"调度模式归纳"工作）                       
  - 新情境下没有近邻案例时只能退化成默认模式                      
  - 案例库一变，决策器要重训                                                       
                                                                                   
  适合什么研究目标                                                                 
                                                                                   
  保守路线，最接近 Memento。论文叙事是"我们把 Memento 的 CBR                       
  应用到了控制类任务的调度层"。增量适中，方法成熟，风险低。
                                                                                   
  ---                                                             
  方向 2：Imitation-Decision（模仿学习式决策）
                                              
  案例库的形态
                                                                                   
  每条 case 是完整调度轨迹：(情境序列, 调度行为序列,                               
  结果)。情境序列就是执行过程中关键时间点的视觉/状态摘要，调度行为序列就是"在 step 
  0 喂了什么 prompt、step 137 换成什么、step 280 又换成什么"。                     
                                                                  
  决策器是什么

  一个小型序列模型（轻量 Transformer 或者                                          
  GRU），输入当前情境（也可以包含一段历史），输出下一个调度行为（喂什么
  prompt、是否切换、是否放弃）。运行时完全不检索案例库，案例库只是它的训练数据。   
                                                                  
  训练目标

  - 行为克隆 (BC)：从成功 case 学"在这种情境下应该输出这个调度行为"                
  - 负样本对比：失败 case 的同类情境-行为对作为负例
  - 进阶可加 reward-weighted BC，让"少步成功"的 case 权重更大                      
                                                                                   
  案例库与决策器的耦合                                                             
                                                                                   
  弱耦合 + 强抽象。案例库一旦训完决策器就可以丢掉（在运行时），但案例库扩展后要重新
  训练决策器。决策器把案例库的知识全部"内化"成参数。
                                                                                   
  关键挑战                                                        

  - 数据量需求大（BC 是 sample-efficient 但要够多样化的轨迹）                      
  - 案例库失去"可解释复用"特性——你不能再说"这条决策是参考了哪条历史案例"
  - 决策器是黑盒，调试难                                                           
                                                                  
  适合什么研究目标                                                                 
                                                                  
  激进路线。论文叙事是"我们用历史经验训出了一个轻量级调度策略"。强调的是"训出来的决
  策器能力"，案例库变成幕后英雄。这条路对用户你的项目可能最不适合——因为你说想要"借
  鉴复用"，BC 范式在运行时是直接生成而非借鉴。                                     
                                                                  
  ---
  方向 3：Retrieval-Augmented Decision（检索增强决策）

  案例库的形态

  每条 case 是 (情境, 调度行为/调度模式, 结果, 摘要文本)，情境用 embedding         
  存（便于近邻检索），调度行为也用结构化字段存。case
  内容相对原子——不是完整长轨迹，而是"在某个调度决策点上做了什么"。                 
                                                                  
  决策器是什么                                                                     
   
  一个条件决策网络：输入是 (当前情境, 检索到的 TopK                                
  案例)，输出是下一步调度行为。检索是决策的输入条件，不是决策的输出。
                                                                                   
  可以这样理解：决策器是一个"看着 K 条参考资料做选择"的学生——参考资料（案例）摆在眼
  前，但最终怎么决策还是它自己学到的。
                                                                                   
  训练目标                                                                         
   
  端到端训练。损失是"决策器输出的调度行为最终带来的成功率"。可以用：               
  - 离线 RL（CQL / IQL 等）从案例库做轨迹回放                     
  - 或者 行为克隆 + retrieval 增强：先对成功 case 做                               
  BC，但训练样本中显式包含"为什么参考了这些 case"的检索结果       
                                                                                   
  案例库与决策器的耦合                                            
                                                                                   
  强耦合 +                                                                         
  强抽象。案例库的每次更新会立即影响决策器的输入（因为运行时检索的池变了），但决策 
  器参数本身不需要重训太频繁。这是兼具"案例库可解释扩展"和"决策器能泛化"的折中方案 
  。                                                              

  关键挑战                                                                         
   
  - 决策器要学会"什么时候相信案例、什么时候忽略案例"，这是 retrieval-augmented     
  系列的经典难题                                                  
  - 训练时要构造"案例池"——给每个训练样本配套它当时能检索到的                       
  cases，工程复杂度上去了                                                          
  - 离线 RL 在 Minecraft 这种长程稀疏奖励任务上有挑战
                                                                                   
  适合什么研究目标                                                                 
                                                                                   
  综合路线。这正对应你之前认可的 C 路线（学习式调度器）。论文叙事最丰富——既有"案例 
  库设计"的方法贡献，又有"参数化决策器"的方法贡献，还有"两者如何耦合"的系统贡献。我
  个人觉得这个方向跟你的诉求最契合，因为它同时满足你说的"借鉴复用"（运行时仍然在查 
  案例库）和"训练决策器"（决策器是真的在学）。

  方向 4：Hierarchical Skill-Decision（分层技能式决策）
                                                                                   
  案例库的形态
                                                                                   
  两层存储：                                                                       
  - 底层是"原子技能库"：从所有调度轨迹里挖掘出 N
  个频繁出现的"调度子模式"（比如"环视寻找资源"、"贴近目标后切换 prompt"、"收集 1   
  个后重新评估"），每个技能是一个抽象单元                                       
  - 上层是案例：每条 case 是 (情境, 用了哪些技能 + 拼装顺序, 结果)                 
                                                                  
  决策器是什么                                                                     
                                                                                   
  两层决策器：                                                                     
  - 上层选择器：决定当前情境下应该用哪几个技能、怎么拼                             
  - 下层执行器：把选定的技能落到具体的 STEVE-1 prompt                              
  序列上（这一层可以是规则化的，也可以是参数化的）   
                                                                                   
  训练目标                                                        
                                                                                   
  - 技能挖掘：聚类/序列模式挖掘（prefix span 等算法）从历史轨迹里找出现频率高 +    
  与成功正相关的子模式                                                             
  - 上层决策器：跟方向 1 类似的检索/分类训练，但 label 是"该用什么技能组合"        
  - 下层执行器：可以靠模板填充，简单可靠                                           
   
  案例库与决策器的耦合                                                             
                                                                  
  中等耦合 + 极强抽象。技能本身就是 case 的"压缩字典"，决策器在技能空间上学习，case
   只是技能的来源。                                               
                                                                                   
  关键挑战                                                                         
   
  - 技能挖掘是一个独立的研究问题，做不好整个体系崩                                 
  - 技能的粒度不好定（粒度太粗失去表达力，粒度太细回到原始轨迹）  
  - 工程量是四个方向里最大的                                                       
                                                                  
  适合什么研究目标                                                                 
                                                                  
  论文卖点路线。如果你要追求"这篇论文的方法图最好看、最有可解释性"，分层技能是最有 
  冲击力的——你可以画出"我们发现 STEVE-1 调度可以被分解为 7 个核心技能，agent
  学会的是技能的拼装规律"这种图。但工程代价大，且 milestone                        
  要切得很清楚才能稳推进。

  一些供你判断的关键问题
                                                                                   
  不需要立刻选定方向，但希望你先在以下几个问题上给我倾向，便于下一轮聚焦：
                                                                                   
  1. 决策器的"决策对象"你倾向哪种：                                                
    - (a) 选 case → 直接套用 case 的调度模式（方向 1 / 方向 4 的上层）             
    - (b) 看 case 后再生成调度行为（方向 3）                                       
    - (c) 完全内化 case 的知识，运行时不查（方向 2）                               
  2. 运行时是否要保留"查案例库"这个动作？                                          
    - 要 → 方向 1 / 3 / 4                                                          
    - 不要（接受决策器变黑盒）→ 方向 2                                             
  3. 你能否接受先做一次"调度模式 / 技能"的人工或半自动归纳？                       
    - 能 → 方向 1 / 4 可行                                                         
    - 不能 → 方向 2 / 3 更合适                                                     
  4. 论文层面你最想强调的是哪一面：                                                
    - 案例库的设计 → 方向 1 / 4                                                    
    - 决策器的能力 → 方向 2 / 3                                                    
    - 两者的协同 → 方向 3 是最佳载体
